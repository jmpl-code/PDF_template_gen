# Retrospective — Epic 3 : Export EPUB Kindle

## Synthèse

Epic compact : ajout du back-end EPUB via Pandoc, puis dual export PDF + EPUB en une commande. 2 stories, mais haute valeur livree car double le format de sortie sans casser le pipeline existant.

## Ce qui a bien fonctionne

- **Reutilisation maximale du AST** — `render_epub()` consomme le meme `BookNode` que `render_pdf()`. Aucun duplication de logique de parsing. C'est la promesse architecturale du compilateur (front-end → IR → back-ends multiples) tenue dans la realite.
- **Pattern `external.py` reutilise tel quel** — l'appel a Pandoc passe par `run_external()` exactement comme Typst. Zero exception, zero workaround.
- **Story 3.1 : convertion AST → Markdown intermediaire (au lieu de HTML direct)** — decision pragmatique : Pandoc gere mieux le markdown intermediaire que tout HTML genere a la main. Permet aussi de reutiliser la TDM Pandoc native.
- **Fallback image pour tableaux > 4 colonnes via Matplotlib** — gracieux, pas de crash, et aligne avec la philosophie "warnings clairs, jamais d'echec silencieux". L'utilisateur recoit un PNG embedded dans l'EPUB.
- **Story 3.2 : echec EPUB graceful** — si le rendu EPUB plante, le PDF est quand meme produit avec un warning. Comportement specifie dans l'epic et tenu : `try/except RenderError` autour de `render_epub()` dans `pipeline.py`.
- **Aucune nouvelle dependance ajoutee en Story 3.2** — la story d'integration s'est limitee a etendre `organize_output()` avec un parametre `epub_path: Path | None = None`. Surface minimale.
- **Artefacts intermediaires isoles dans `_epub_build/`** — pas de pollution du dossier de sortie principal. Pattern reutilisable.

## Ce qui a coince

- **CSS Kindle minimal** — basique : serif, line-height, headings, img max-width. Suffisant pour MVP1 mais pas customisable via tokens (ce n'est venu qu'avec Story 4.1). Aurait pu etre flagge comme dette explicite.
- **Asymetrie LLM-judge PDF/EPUB toujours non resolue** — la boucle LLM ne touche que le PDF (vu en architecture, decision 10). Si Story 4.x ajuste les tokens, l'EPUB doit etre re-rendu. Pas adresse en Epic 3.
- **Pas de test de rendu reel EPUB sans Pandoc** — les tests mockent `run_external`. Le golden file n'est pas une option pour EPUB (binaire ZIP). On valide donc le contrat (commandes Pandoc passees), pas le contenu binaire. Acceptable, mais limite.

## Lecons retenues

1. **Un IR bien concu paye lors du second back-end.** Le AST + ResolvedTokenSet ont permis a Story 3.1 de livrer EPUB en quelques heures de design + impl, exactement comme l'architecture le promettait.
2. **Test du contrat (commandes externes) > test du binaire** quand le binaire est opaque (EPUB ZIP). Pas un compromis, juste une pratique adaptee au format.
3. **Echec graceful avec warning clair** est une regle qui paye sur le terrain. Story 3.2 le met en pratique : un EPUB qui plante ne doit jamais bloquer le PDF.
4. **Une fonction d'extension acceptant `Path | None`** est plus simple a tester et plus retro-compatible que de creer une nouvelle fonction parallele. `organize_output(epub_path=None)` > `organize_output_with_epub()`.

## Dette technique heritee

- **CSS EPUB pas pilote par tokens** en Story 3.1 — corrige partiellement en Story 4.1 (CSS dynamique avec ratios calcules), mais reste minimal.
- **Pas de tests de rendu binaire EPUB** — on teste le contrat, pas le contenu reel. Si Pandoc change son comportement, on ne le verra pas avant la prod.
- **Asymetrie LLM-judge PDF vs EPUB** — non adressee, sera a clarifier en Epic 6 (LLM-judge manuel).
