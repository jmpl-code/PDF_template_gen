## Synthèse

Epic de personnalisation : systeme de design tokens YAML avec assertions, classe `business-manual` avec defaults, balises semantiques (`:::framework`, `:::callout`, `:::chapter-summary`), numerotation romaine/arabe et escape hatch Typst. C'est l'epic qui a transforme BookForge d'un compilateur fige en un outil customisable.

## Ce qui a bien fonctionne

- **TokenSpec + TOKEN_REGISTRY avec assertions min/max/source** (Story 4.1) — pattern qui a tenu sur tout l'epic. Les warnings explicites sur valeurs hors bornes ("la borne et la source de reference") sont exactement ce que demande NFR6 ("zero crash silencieux").
- **`ResolvedTokenSet` Pydantic immuable** — passe en parametre aux deux renderers (PDF et EPUB) sans copie ni mutation. Triplet (AST, ClassConfig, ResolvedTokenSet) tenu en pratique.
- **Backward compat parfaite a chaque story** — `tokens` est `| None` partout, le pipeline pre-Epic 4 fonctionne sans changement. 0 regression sur 184 → 223 tests durant tout l'epic.
- **Plugin markdown-it-py natif (Story 4.3) — pas de `mdit-py-plugins`** — decision d'eviter une nouvelle dependance pour respecter NFR13 (5/5 deps). Pattern container ecrit a la main, fonctionne, et le budget deps est preserve.
- **3 noeuds AST semantiques avec exactement le pattern Story 2.2** — `FrameworkNode`, `CalloutNode`, `ChapterSummaryNode` ont le meme contrat (`frozen`, `source_file`, `line_number`). Lecon : un bon pattern de socle se reutilise sans douleur 4 epics plus tard.
- **Story 4.4 : 2 FRs distincts groupes en 1 story** — FR14 (numerotation) et FR27 (escape hatch) n'ont pas d'interaction et sont chacun ~ 2-3h de travail. Les grouper a evite une story sous-dimensionnee. Pattern a reutiliser quand 2 FRs sont small et indep.
- **Escape hatch Typst injecte verbatim** (Story 4.4) — decision explicite : pas d'echappement, c'est tout l'interet d'un escape hatch. Documente comme tel dans les anti-patterns.
- **Erreur enrichie quand `typst_raw` present** — si Typst plante et qu'un escape hatch est defini, le message le mentionne explicitement. UX d'erreur soignee.
- **`mypy strict` sur src/bookforge clean a chaque commit** — aucun raccourci `# type: ignore`.

## Ce qui a coince

- **Assertion regex Pydantic pour `document_class`** (Story 4.2) — la regex `^[a-z][a-z0-9]*(-[a-z0-9]+)*$` est tatillon. Acceptable mais devrait peut-etre etre warning + fallback plutot qu'erreur dure. A reconsiderer.
- **CSS EPUB dynamique via ratios calcules** (Story 4.1) — fonctionne mais le calcul `heading_size / font_size` est fragile : si l'auteur met `font_size: 0`, ca casse. Story 4.1 gere via `min/max` du registre, mais le couplage entre CSS et tokens est plus implicite que cote PDF.
- **`generate_typst_no_front_matter_without_config` adapte 2 fois en Story 4.4** — Le test verifiait `numbering: none` avant le changement. Adapte a `numbering: "i"`. Symptome : les tests qui s'appuient sur des chaines specifiques du template sont fragiles. Golden files seraient plus robustes pour ce cas.
- **Story 4.3 : 26 tests ajoutes mais pas de fixture or refonte de l'integration de bout-en-bout** — la fixture `with_semantic/` est minimale. On teste les 3 noeuds individuellement mais peu le mix avec les autres types de contenu.
- **Code review Story 4.1 a genere des fixes posteriors** (commit `9ee0212e`) — symptome : la story 4.1 a ete commitee avant le code-review. Process : faire le code-review AVANT le commit `feat: implement` pour eviter les commits `fix: address review`.
- **`epic-4` reste `in-progress` dans `sprint-status.yaml`** alors que ses 4 stories sont toutes `done` — la transition manuelle n'a pas ete faite. A corriger dans cette retro.

## Lecons retenues

1. **Backward compat via `| None` est le pattern le plus simple et le plus testable** quand on ajoute un parametre optionnel a une fonction de renderer.
2. **Eviter d'ajouter une dependance pour une feature optionnelle** — meme quand la dep est "officielle" (`mdit-py-plugins` est de la meme team que `markdown-it-py`). 5/5 deps reste 5/5.
3. **Grouper 2 FRs minuscules dans une seule story** est un pattern valide quand ils sont independants. Eviter pour des FRs qui se chevauchent.
4. **Escape hatch = injection verbatim sans validation** — c'est par definition. Documenter explicitement dans les anti-patterns.
5. **Adapter les tests existants quand on change un comportement public est un signe que les assertions sont trop specifiques.** Golden files > sous-strings.
6. **Story 4.4 a beneficie enormement du contexte preserve dans la story spec** (Dev Notes detaillees, points d'injection avec numero de ligne). Pattern `bmad-create-story` paye.
