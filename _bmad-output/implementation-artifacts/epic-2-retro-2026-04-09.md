## Synthèse

Le plus gros epic du MVP : du Markdown a un PDF complet avec front matter, TDM, pages de garde de chapitre, headers/footers, images, couverture, metadonnees KDP, CLI fonctionnel. C'est l'epic ou tout le pipeline `parse → render → export` est ne.

## Ce qui a bien fonctionne

- **Architecture en compilateur respectee** — front-end (parser) → middle-end (AST) → back-end (Typst) → post-processing (export). Le decoupage epic en stories suit exactement ce flux, ce qui a evite les couplages.
- **AST avec frozen dataclasses + `source_file`/`line_number` partout** — pattern etabli en Story 2.2, repris sans deviation dans toutes les stories suivantes (4.3 a ajoute 3 noeuds avec exactement le meme pattern).
- **Template Typst inline (vs fichier externe)** — decision Story 2.3 d'embarquer le template comme constante Python `_BASE_TEMPLATE`. A evite les problemes de resolution de chemin cross-platform et facilite le test golden file.
- **`escape_typst()` centralise** — toute interpolation utilisateur passe par lui. Aucun bug d'echappement signale dans tout l'epic.
- **Test golden file pour la stabilite du `.typ` genere** — Story 2.2 a etabli `tests/fixtures/golden/minimal.typ`. Permet de detecter immediatement toute modification involontaire du rendu.
- **Pattern `try/except RenderError` graceful sans crash silencieux** — Story 2.6 introduit `RenderError` si image hors repertoire ; le pattern est remonte propre jusqu'au CLI.
- **Refactoring opportun en Story 2.5** — la numerotation et le reset compteur ont ete deplaces de `_render_front_matter()` vers `_render_running_headers()` pour eviter l'ecrasement par le second `#set page()`. Refactoring necessaire (vu en Story 2.5) plutot que workaround.
- **CLI complet en Story 2.9** — pas de stub : options reelles, exit codes, logging structure, callbacks de progression. La story finale de l'epic a mis le bow sur le paquet.

## Ce qui a coince

- **Decision build_dir ambigue en Story 2.9** — l'image `.typ` resout les chemins en relatif via `Path.relative_to`. Le build dir doit donc etre la racine du livre (book_root), pas un sous-dossier separe. Decouvert en cours d'implementation, a force de revoir l'architecture du pipeline. **Lecon :** quand un module impose une contrainte de filesystem, la documenter explicitement.
- **Code review Gemini sur Story 2.3 — 6 findings dont 2 critiques** — cf. commit `94b48aff`. Findings : `source_file`/`line_number` manquants sur `BookNode`, et autres. Lecon : faire le code-review systematiquement sur les stories de socle (les premieres d'un epic) ou les bugs ont l'effet le plus large.
- **9 stories dans un seul epic** — c'est beaucoup. Le risque d'oubli (un AC manque, une regression) augmente. Pour l'epic 2 c'est passe, mais a future eviter ce volume.
- **Story 2.4 a deplace de la logique en Story 2.5** — la numerotation initialement geree par `_render_front_matter()` a du etre deplacee. Symptome : la separation des responsabilites n'etait pas claire au debut. Pas dramatique car corrige rapidement.

## Lecons retenues

1. **Test golden file > assertions sur sous-strings** quand on genere du code (Typst, CSS, HTML). Plus stable, plus diff-able, plus simple a maintenir.
2. **Frozen dataclasses + `source_file`/`line_number` obligatoires** sur tout noeud AST. Permet le logging contextualise et c'est devenu un pattern projet.
3. **Centraliser les fonctions d'echappement** des le debut. Un seul `escape_typst()`, jamais inline.
4. **Code review systematique sur les stories de socle** — les bugs d'architecture coutent cher a corriger plus tard.
5. **Eviter les epics > 6-7 stories** — le risque d'oubli grandit non-lineairement.
