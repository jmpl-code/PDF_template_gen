## Synthèse

Epic fondationnel pur : initialisation du projet Python (uv), structure des modules, hierarchie d'erreurs, runner subprocess uniforme, CI GitHub Actions. Aucune fonctionnalite metier livree mais toutes les fondations en place pour le pipeline.

## Ce qui a bien fonctionne

- **Cadrage strict du scope** — chaque story s'est arrete a son perimetre, sans deborder. Story 1.1 = squelette uv + tests d'import. Story 1.2 = modules + erreurs + `run_external`. Story 1.3 = CI. Pas de melange.
- **Conventions etablies des le depart et tenues sur tout l'epic** :
  - `pathlib.Path` partout, jamais `os.path`
  - `logging.getLogger("bookforge.<module>")`, jamais `print()`
  - Messages utilisateur en francais via exceptions, logs internes en anglais
  - `subprocess` sans `shell=True`
- **CI cross-platform multi-version Python** — la matrice 3 OS x 4 versions Python a ete mise en place tres tot, ce qui a evite de payer le cout de la portabilite plus tard
- **Hierarchie d'erreurs typee avec exit codes differencies** (`InputError`=1, `RenderError`=2, `LLMError`=3) — pattern utilise sans accroc dans tous les epics suivants
- **`run_external()` centralise** — convention etablie tres tot, reutilisee par les renderers PDF, EPUB, et cover sans variation

## Ce qui a coince

- **Conflit d'outillage avec le repo BMad parent** — `ruff check .` captait les fichiers Python du projet BMad existant. Resolution : ajouter `include` dans `[tool.ruff]` pour limiter au scope BookForge. Lecon retenue : un projet imbrique dans un workspace BMad doit explicitement scoper ses linters.
- **Defaut uv vs requis** — `uv init --lib` genere `requires-python` a la version locale (3.13) ; il a fallu corriger a `>=3.10` pour respecter l'architecture. Et `uv_build` a ete remplace par Hatchling manuellement.
- **22 fichiers placeholder cree en Story 1.2** — choix discutable : pollue la PR initiale et donne une fausse impression de progres. Avantage : a permis aux stories suivantes de remplir des stubs au lieu de creer de nouveaux fichiers (pattern utile en Story 4.3 par exemple).

## Lecons retenues

1. **Etablir les conventions de code dans le tout premier epic** et les tenir sans exception — paye des le 2e epic.
2. **CI cross-platform des le J1** — moins cher de payer le cout maintenant que de chasser un bug Windows en Story 2.6.
3. **Hierarchie d'erreurs > exceptions ad-hoc** — exit codes differencies sont utiles pour le scripting CLI.
4. **Stubs vs lazy creation** — les fichiers stubs ont ete utiles pour les stories qui s'y sont insere (4.3 sur `parser/semantic.py` et `ast_nodes/semantic.py`), mais creent du bruit dans `git status`. A reflechir pour les futurs projets.
